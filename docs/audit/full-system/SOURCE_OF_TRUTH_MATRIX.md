# Source of Truth Matrix

Kolonner: **hvor sannheten ligger i dag** (konkret fil/modul) vs **hvor den bør ligge** (mål). Alle «i dag»-felt er sporet til kode.

| Konsept | Sannhet i dag (bevis) | Bør ligge (mål) |
|---------|------------------------|-----------------|
| **Side** (CMS-side som entitet) | **Supabase** tabeller `content_pages` / tilknyttede rader; dokumentert i `lib/cms/public/loadLivePageContent.ts` linje **1–4**: *«`content_pages` (published) + `content_page_variants` … DB is source of truth»*. | Én eksplisitt **domain-modell** (typer + repos) som alle API-er leser fra; genererte DB-typer fra `supabase gen types` i stedet for `LoosePublicTable` (`lib/types/database.ts` **15–27**). |
| **Blokk** | **JSON** i side-body + **registry** for rendering: `lib/public/blocks/renderBlock.tsx` bruker `resolveBlockForEnterpriseRender` / `EnterpriseLockedBlockBridge` (**44–91**); modell-typer i `lib/cms/model/blockTypes` (importert linje **3**). | Én **valideringspipeline** (Zod) per blokk-type ved skriving, ikke bare ved render; sentral manifest-fil som lister blokker og versjon. |
| **Preview data** | **Samme parse/normaliser** som prod: `app/(backoffice)/backoffice/preview/[id]/page.tsx` **27–30** (*«identical rendering pipeline … parseBody → normalizeBlockForRender → renderBlock»*); `PREVIEW_ENV = "staging"` **8–10**. `PreviewCanvas.tsx` importerer `renderBlock` (**9**, **166**-området). `loadLivePageContent` støtter variant via `GetContentBySlugOptions` (`lib/cms/public/loadLivePageContent.ts` **25–28**). | **Identisk** kodebane for `renderBlock` + env-flag `staging`/`prod` (allerede delvis); **må** sikres med **grønne** parity-tester (f.eks. fiks `tests/motion/motionSystemProof.test.ts` **102–110**). |
| **Published render** | **Offentlig** lesing: `loadLivePageContent` → `getContentBySlug` → `parseBody` (`lib/cms/public/loadLivePageContent.ts` **25–37**). Komponenter: `components/cms/CmsBlockRenderer.tsx` (`grep` viser import av `renderBlock`). | **Samme** som rad over — publisert skal være **deterministisk** ut fra published variant i DB; ingen «skjult» klient-cache av sannhet. |
| **Auth / rolle** | **Server `getAuthContext`** (`lib/auth/getAuthContext.ts` **20–41**): `role`, `company_id`, `location_id`, `rid`. **Landing/allowlist:** `lib/auth/role.ts` `landingForRole` / `allowNextForRole` (**17–35**). **Post-login:** `app/api/auth/post-login/route.ts` (bruker `resolvePostLoginTarget` — se fil). | **Én** server-sannhet (allerede intent); **ikke** dupliser rolle i klient for redirect. |
| **Tenant** | **profiles.company_id** (og `location_id` der relevant) — `AuthContext` (**26–27**); `AGENTS.md` C3 krever server-sannhet. API: `src/lib/guards/assertCompanyActiveApi.ts` kaller `assertCompanyActive` fra `@/lib/guards/assertCompanyActive` (**4–5**). | **Samme** — **alle** queries må filtrere på `company_id`; **må** verifiseres per API-route (stikkprøve, ikke 314 verifisert i audit). |
| **Settings** | **Rad** i `system_settings`: `getSystemSettingsSafe` / `getSystemSettings` (`lib/system/settings.ts` **89–119**); cache `getCachedSettings` (`lib/settings/cache.ts` **24–31**). **Kill switch** via `enforceSystemGate` (`lib/system/enforcement.ts` **22–41**). | **Samme tabell**; forbedre **type-sikkerhet** og **observability** (erstatt `console.error` **93–106** med strukturert logg). |
| **Media** | **Ikke fullstendig kartlagt** i denne matrisen; APIflate finnes: `app/api/backoffice/media/**` (stikkprøve fra grep). **Kontrakt-dok:** `docs/MEDIA_API_CONTRACT.md` (finnes i repo). | **Én** media-tjeneste + metadata i DB; **les** `docs/MEDIA_API_CONTRACT.md` og implementer **én** klient. |
| **Routing / slug** | **Slug** fra `content_pages` via `getContentBySlug` (`lib/cms/public/getContentBySlug.ts` — importert av `loadLivePageContent` linje **8–9**). Next **App Router** mapper URL → `app/(public)/...` layouts. | **Én** slug-normaliseringsfunksjon brukt både ved lagring og ved resolve (fail-closed på kollisjon). |
| **SEO** | **Redaksjonelt** (`ContentSeoPanel` m.fl. under `backoffice/content/_components/` — fra audit); **metadata** helpers i `lib/cms/public/cmsPageMetadata.ts` (referert i `grep`/`lib/cms`). | **Skille** innhold (tekst) fra presentasjon; metadata lagret i DB-felter, ikke kun i klient-state. |
| **Workflow / publish state** | **Versjoner og releases:** `lib/backoffice/content/pageVersionsRepo.ts` (navn fra audit); API-ruter `app/api/backoffice/content/**` inkl. `pages/[id]/workflow`, `releases`. **Kommentar** i `loadLivePageContent.ts` linje **3–4**: *«publish promotes preview → prod in backoffice»*. | Eksplisitt **state machine** (draft → review → published) med **én** rad/tabell som autoritativ status; **ikke** spredt boolean i JSON uten migrasjon. |

## Merknader

- **Sanity** (`studio/sanity.config.ts`, `studio/lunchportalen-studio/sanity.config.ts`) er **ikke** primær SoT for marketing-sider — se `loadLivePageContent` som peker på **Supabase** (`content_pages`). Sanity brukes **til** bl.a. ukeplan (`lib/sanity/*.ts` — separat spor).
- **Dualitet** Sanity vs Supabase er **bevisst** i arkitekturen, men **krever** dokumentert grense — ellers **to sannheter** for «innhold» avhengig av kontekst.
